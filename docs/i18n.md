# Internationalisation (i18n)

> Translation standards, plugin choice, URL strategy, and REST API language filtering.

---

## Plugin: Polylang (free)

Install via Composer (WPackagist):

```bash
composer require wpackagist-plugin/polylang
```

- Danish is the default language (no URL prefix)
- English is the secondary language (`/en/` prefix)
- Polylang handles slug translation per language — CPT slugs are always registered in English in code (e.g. `event`), and Polylang maps the Danish URL slug (`begivenhed`) at the language layer
- No hard-coded Danish strings in PHP or JS — everything goes through translation functions

---

## URL Structure

| Language | Event archive | Single event |
|---|---|---|
| Danish (default) | `/begivenheder/` | `/begivenhed/my-event/` |
| English | `/en/events/` | `/en/event/my-event/` |

Slug translations are configured in Polylang settings, not in PHP.

## PHP — Translation Functions

Always use WP translation functions. Text domains: `vandrekalender-events` (plugin), `vandrekalender-theme` (theme).

```php
// Basic string
__( 'Walking events in Denmark', 'vandrekalender-events' )

// Escaped for HTML output (use this in templates)
esc_html__( 'Walking events in Denmark', 'vandrekalender-events' )

// With echo
esc_html_e( 'Walking events in Denmark', 'vandrekalender-events' )

// With variable substitution
sprintf(
    /* translators: %s: event title */
    esc_html__( 'Sign up for %s', 'vandrekalender-events' ),
    esc_html( $event_title )
)
```

Never output a bare English string — always wrap in a translation function.

---

## JavaScript — Translation Functions

Use `@wordpress/i18n` (already a dependency via `@wordpress/scripts`).

```js
import { __, sprintf } from '@wordpress/i18n';

// Basic string
__( 'Filter events', 'vandrekalender-events' )

// With variable
sprintf( __( 'Showing %d events', 'vandrekalender-events' ), count )
```

Register the script translations in PHP so WP loads the `.json` translation file:

```php
wp_set_script_translations( 'your-script-handle', 'vandrekalender-events' );
```

## REST API — Language Filtering

Polylang adds a `lang` parameter to all REST API requests.

```
GET /wp-json/vandrekalender/v1/events?lang=da
GET /wp-json/vandrekalender/v1/events?lang=en
```

The custom REST API endpoints in `class-event-rest-api.php` must pass `lang` through to `WP_Query` so Polylang's query filter applies. Without this, the endpoint returns events from all languages.

```php
// In get_events() — add lang param support
if ( $request->get_param( 'lang' ) ) {
    $args['lang'] = sanitize_text_field( $request->get_param( 'lang' ) );
}
```

Default behaviour when no `lang` param is passed: return the default language (Danish).

## Generating Translation Files

```bash
# Extract strings from PHP into a .pot file
wp i18n make-pot wp-content/plugins/vandrekalender-events \
  wp-content/plugins/vandrekalender-events/languages/vandrekalender-events.pot

# Extract strings from JS build output
wp i18n make-json wp-content/plugins/vandrekalender-events/languages
```

`.pot` file is committed to the repo. Compiled `.mo` files are gitignored.
